Product Team Workflow

Step 1: Initialization

Product team member opens Claude Code in a repository (or specifies target repo)

Step 2: Create Work Item

Run /create work item command with either:

  • An idea (Claude will interview to flesh out details)
  • A file reference (Claude will analyze and create work item)
  • A prototype or documentation
Step 3: Template Selection

Choose work item type: Feature, Bug, or Defect (with appropriate templates)

Step 4: Priority Assignment

Set priority level (maps to standard GitHub labels for reporting)

Step 5: Validation

Issue Validator agent reviews acceptance criteria for:

  • Template compliance
  • Testability for engineering
  • Completeness
Step 6: Preview & Workshop

Review the generated issue, make adjustments, or request changes from Claude

Step 7: T-Shirt Sizing

Claude suggests Small/Medium/Large based on complexity (can be edited)

Step 8: Design Check (Optional)

If design needed, mark as "needs design" - routes to design team queue

Step 9: Mark Ready

When complete, mark as "ready" - moves to engineering review queue

Engineering Team Workflow

Step 1: Review Queue

Run /dev review command to see work items marked "ready"

Step 2: Validation

Same Issue Validator runs again to ensure quality gates are met

Step 3: Grooming

Engineer reviews work item and can:

  • Accept: Approve and move to accepted queue
  • Reject: Provide feedback, adds engineering review comment, removes "ready" label
  • Edit First: Make minor adjustments before accepting
Step 4: Size Confirmation

Validate or adjust the suggested t-shirt size

Step 5: Select Work

Engineers pull from accepted queue (no assignment - team self-organizes)

Step 6: Implementation

Engineer works with Claude Code to implement the solution

Step 7: Completion

Testing, PR review, and deployment (future plugin enhancements may automate testing validation)

üí° Implementation Best Practices

For Product Teams

  • Run Claude Code from within the target repository for best context
  • Use /create work item with file references for complex features (PRDs, prototypes, etc.)
  • Let Claude interview you to flesh out ideas - it will ask good clarifying questions
  • Don't skip the validation step - let the Issue Validator improve quality
  • Workshop acceptance criteria until they're truly testable
  • Use the product reference folder for design systems, API docs, and prototypes
  • Mark work items "needs design" early if design input required
  • Be prescriptive about business logic, data mapping, and API endpoints when you have specific requirements

For Engineering Teams

  • Use Opus 4.5 as default model for best results
  • Run /dev review regularly to stay on top of new work items
  • Reject with specific, actionable feedback rather than accepting unclear work
  • Validate or adjust t-shirt sizing based on actual technical complexity
  • Pull work from accepted queue based on interest/expertise (self-organization)
  • Treat acceptance criteria as Given/When/Then test scenarios
  • Use split command if work item scope is too large after investigation
  • Reference product reference folder for design patterns and technical constraints

For Design Teams

  • Set up label subscription or team assignment for "needs design" items
  • Check product reference folder for existing prototypes and context
  • When design complete, mark as "design ready" to return to requirements phase
  • Can add design artifacts directly to issue or check into reference folder

Repository Setup

  • Create product reference folder in each repo as standard practice
  • Consider adding architecture documentation listing dependencies
  • Include product "cover page" describing system touchpoints
  • Set up SDLC project board at org level to see all work across repos
  • Ensure GitHub CLI is installed and authenticated for all team members
