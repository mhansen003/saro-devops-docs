TRACT 01

Dependency Management Strategy

Structuring Complex Projects to Minimize Dependencies - Case Study: Conditions Portal (43 Tickets)

The Challenge

The Big Project Problem

When building a large application like the Conditions Portal, we face several challenges:

  • Many Moving Parts: Multiple roles (Processor, Underwriter, Borrower), complex workflows, and interdependent features
  • Dependency Hell: Features that can't be built until other features exist
  • Blocked Progress: Engineers waiting on foundational work before starting their tasks
  • Integration Risk: Building everything in isolation and hoping it works together
  • Testing Nightmare: Can't test role-specific features without the underlying platform
The Core Question

How do we break down a large project into manageable work items that:

  • Minimize dependencies between tickets
  • Allow parallel development where possible
  • Provide value at each stage
  • Enable incremental testing and validation

The Solution: Three-Tier Dependency Strategy

The Approach

Structure work items in three tiers, each building upon the previous:

  1. Tier 1: Platform Foundation - The base grid/table with read-only data display, no features (1 ticket)
  2. Tier 2: Role Scaffolding - Role-specific UI shells, navigation, and permissions (3 tickets - one per role)
  3. Tier 3: Advanced Features - Individual features within each role context (39 tickets across all roles)

Why This Works

  • Single Point of Truth: Platform ticket establishes data models, API contracts, and UI framework that everyone builds on
  • Parallel Role Development: Once platform is done, multiple teams can work on different roles simultaneously
  • Feature Independence: Within a role, features have minimal dependencies on each other
  • Incremental Value: Each tier delivers working software that can be deployed and tested
  • Clear Blockers: Dependencies are explicit and easy to track

Key Insight

Notice how dependencies flow downward through tiers, but features within a tier can be developed in parallel. This structure maximizes team velocity while minimizing blocked work.

Understanding the Flow

  • Tier 1 must be completed before Tier 2: The platform foundation (CP-001) establishes the data models, API contracts, and base UI that all roles depend on.
  • Tier 2 enables parallel work across roles: Once scaffolding tickets (CP-100, CP-200, CP-300) are complete, teams can work simultaneously on Processor, Underwriter, and Borrower features.
  • Tier 3 builds on completed role features: Individual features (CP-101 through CP-404) have minimal cross-dependencies, allowing flexible prioritization and parallel development.

Success Criteria for This Strategy
  • Engineers never say "I'm blocked waiting on X"
  • Multiple teams can work on different roles simultaneously
  • Each tier delivers deployable, testable software
  • Dependencies are explicit and tracked in GitHub
  • Product team can prioritize features within a role independently

Implementation Best Practices

Dependency Management Guidelines

  • Explicit Dependencies: Always list "Depends On" and "Blocks" in ticket descriptions
  • GitHub Labels: Tag tickets with tier labels (platform, role, feature) and status labels (blocked, ready, in-progress)
  • Link Related Tickets: Use GitHub's linking syntax (#CP-XXX) to create clickable references
  • Order Matters: Create tickets in dependency order so engineers see the logical flow
  • Ready Gates: Only mark Tier 2 tickets "ready" after Tier 1 is complete and deployed
  • Feature Independence: Within a tier, design features to have minimal cross-dependencies

Ticket Naming Convention

# Format: [Project]-[Tier][Role][Sequence] # Platform (Tier 1): CP-001 to CP-099 Example: CP-001 Conditions Grid Platform # Role Scaffolding (Tier 2): CP-[Role]00 Example: CP-100 Processor Role Shell Example: CP-200 Underwriter Role Shell Example: CP-300 Borrower Role Shell # Features (Tier 3): CP-[Role][Feature] Example: CP-101 Request Documents (Processor) Example: CP-201 Clear Condition (Underwriter) Example: CP-301 View Required Docs (Borrower) Example: CP-400 Activity History (Cross-cutting)

Testing Strategy Per Tier

Tier Testing Focus Acceptance Gate
Platform Can users view data? Is API contract stable? Does grid render correctly? Deploy to staging, have all roles log in and verify data loads
Role Scaffolding Can each role access their view? Are permissions correct? Does navigation work? Have actual users (processor, underwriter, borrower) test their views
Features Does the feature work end-to-end? Happy path + error cases User acceptance testing with real workflow scenarios
