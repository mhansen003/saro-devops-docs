Structuring Complex Projects to Minimize Dependencies

Case Study: Conditions Portal

üéØ The Challenge

The Big Project Problem

When building a large application like Conditions Portal, we face several challenges:

  • Many Moving Parts: Multiple roles (Processor, Underwriter, Borrower), complex workflows, and interdependent features
  • Dependency Hell: Features that can't be built until other features exist
  • Blocked Progress: Engineers waiting on foundational work before starting their tasks
  • Integration Risk: Building everything in isolation and hoping it works together
  • Testing Nightmare: Can't test role-specific features without the underlying platform
üí° The Core Question

How do we break down a large project into manageable work items that:

  • Minimize dependencies between tickets
  • Allow parallel development where possible
  • Provide value at each stage
  • Enable incremental testing and validation

‚úÖ The Solution: Three-Tier Dependency Strategy

The Approach

Structure work items in three tiers, each building upon the previous:

  1. Tier 1: Platform Foundation - The base grid/table with read-only data display, no features
  2. Tier 2: Role Scaffolding - Role-specific UI shells, navigation, and permissions (no business logic yet)
  3. Tier 3: Feature Implementation - Individual features within each role context

Why This Works

  • Single Point of Truth: Platform ticket establishes data models, API contracts, and UI framework that everyone builds on
  • Parallel Role Development: Once platform is done, multiple teams can work on different roles simultaneously
  • Feature Independence: Within a role, features have minimal dependencies on each other
  • Incremental Value: Each tier delivers working software that can be deployed and tested
  • Clear Blockers: Dependencies are explicit and easy to track

Tier 1: Platform Foundation Ticket

#CP-001
Platform Large Priority: High
Conditions Portal - Platform Foundation (Read-Only Grid)
Summary:

Create the foundational platform for the Conditions Portal, including a read-only grid/table that displays condition data for all user roles. This establishes the core infrastructure, data models, API contracts, and UI framework that all subsequent role and feature work will build upon.

Scope:
  • Data model for conditions (id, type, status, assigned_to, due_date, description, etc.)
  • API endpoints: GET /conditions, GET /conditions/{id}
  • Grid component displaying conditions data (sortable, filterable)
  • Basic authentication and session management
  • Top-level navigation structure (placeholder for roles)
  • Empty state handling
  • Responsive layout foundation
Acceptance Criteria:
  • ‚úì User can log in and see the conditions grid
  • ‚úì Grid displays all conditions with columns: ID, Type, Status, Assigned To, Due Date, Description
  • ‚úì Grid supports sorting by any column
  • ‚úì Grid supports filtering by status and assigned user
  • ‚úì Clicking a row shows condition details in a read-only detail panel
  • ‚úì No action buttons or edit capabilities (read-only)
  • ‚úì API returns JSON with consistent structure documented
  • ‚úì Works on desktop and mobile viewports
Out of Scope:
  • Any create, update, or delete operations
  • Role-specific views or features
  • Document upload/download
  • Notifications or alerts
Dependencies: None (Foundation ticket)
Blocks: #CP-100 #CP-200 #CP-300

Tier 2: Role Scaffolding Ticket

#CP-100
Role Medium Priority: High
Processor Role Shell - UI Scaffolding & Navigation
Summary:

Create the Processor role-specific view and navigation shell. This establishes the UI container and action panel framework where processor-specific features will be implemented, but does not include feature business logic yet.

Scope:
  • Processor-specific view of the conditions grid (filtered to show processor-assigned items)
  • Action panel on the right side with placeholder sections for future features
  • Navigation to toggle between "My Conditions" and "All Conditions"
  • Permission checks: user must have "processor" role to access this view
  • Empty states for each action section (e.g., "Document actions will appear here")
Acceptance Criteria:
  • ‚úì Processor sees grid filtered to their assigned conditions by default
  • ‚úì Action panel displays on right side with labeled sections: "Document Requests", "Review Actions", "Decision Actions"
  • ‚úì Each action section shows placeholder text (no functional buttons yet)
  • ‚úì Navigation allows toggle between "My Conditions" and "All Conditions"
  • ‚úì Non-processor users cannot access this view (permission check)
  • ‚úì Selecting a condition highlights it and updates action panel context
Out of Scope:
  • Any functional action buttons (request docs, accept/reject, etc.)
  • API endpoints for processor actions
  • Document upload/download functionality
Depends On: #CP-001 (Platform Foundation)
Blocks: #CP-101 #CP-102 #CP-103

Tier 3: Feature Implementation Ticket

#CP-101
Feature Small Priority: Medium
Processor Feature - Request Documents from Borrower
Summary:

Implement the "Request Documents" feature for Processor role. Processor can select a condition, specify which documents are needed, and send a request to the borrower. Borrower receives notification and sees the request in their view.

Scope:
  • Add "Request Documents" button in Processor action panel (Document Requests section)
  • Modal dialog to select document types and add custom message
  • API endpoint: POST /conditions/{id}/document-requests
  • Update condition status to "Documents Requested"
  • Send notification to borrower (email + in-app)
  • Log action in condition activity history
Acceptance Criteria:
  • ‚úì Given a processor has selected a condition, when they click "Request Documents", then a modal opens with document type checkboxes and message field
  • ‚úì Given the processor submits the request, when the API call succeeds, then the condition status updates to "Documents Requested"
  • ‚úì Given a document request is submitted, when the borrower logs in, then they see the request in their conditions view
  • ‚úì Given a document request is submitted, then the borrower receives an email notification with document list and deadline
  • ‚úì Given a request is submitted, then it appears in the condition's activity history with timestamp and processor name
Data Mapping:
  • Document types: enum (PayStub, BankStatement, TaxReturn, EmploymentLetter, Other)
  • API payload: { condition_id, requested_documents: [{ type, description, due_date }], message }
  • Notification: Use Docs notification service with template "document-request"
Depends On: #CP-100 (Processor Role Shell)
Related: #CP-301 (Borrower must be able to view requests)

üöÄ Implementation Guidelines for Product Team

1. Ticket Naming Convention

# Format: [Project]-[Tier][Role][Sequence] # Platform (Tier 1): CP-001 to CP-099 Example: #CP-001 Conditions Grid Platform # Role Scaffolding (Tier 2): CP-[Role]00 Example: #CP-100 Processor Role Shell Example: #CP-200 Underwriter Role Shell # Features (Tier 3): CP-[Role][Feature] Example: #CP-101 Request Documents (Processor) Example: #CP-201 Clear Condition (Underwriter)

2. Dependency Management Best Practices

  • Explicit Dependencies: Always list "Depends On" and "Blocks" in ticket description
  • Use GitHub Labels: Tag tickets with "blocked" label when dependencies aren't met
  • Link Related Tickets: Use GitHub's linking syntax (#CP-XXX) to create clickable references
  • Order Matters: Create tickets in dependency order so engineers see the logical flow
  • Ready Gates: Only mark Tier 2 tickets "ready" after Tier 1 is complete and deployed

3. What to Include in Each Tier

Tier 1: Platform Foundation
  • Data Models: Define all entities, fields, types, validations
  • API Contracts: Specify endpoints, request/response shapes, error codes
  • UI Framework: Core components (grid, navigation, detail panel)
  • Authentication: Login, session, role detection
  • Read-Only: Display data only, no mutations
Tier 2: Role Scaffolding
  • Role-Specific Views: Filtered grids, specialized layouts
  • Action Panels: Empty containers where features will live
  • Navigation: Role-specific menu items and routes
  • Permissions: Role checks and access control
  • No Business Logic: Just the UI shell, placeholders for actions
Tier 3: Feature Implementation
  • Single Responsibility: One feature per ticket (e.g., "Request Documents" only)
  • Complete Vertical Slice: UI + API + DB for that specific feature
  • Given/When/Then: Use BDD acceptance criteria format
  • Data Mapping: Specify exact field mappings, API payload structure
  • Error Handling: Define what happens when things fail

4. When Features Have Cross-Role Dependencies

Problem: Processor "Request Documents" (#CP-101) requires Borrower "View Requests" (#CP-301)

Solution Options:

  1. Option A - Sequential: Build borrower viewing capability first, then processor request feature depends on it
  2. Option B - Stub First: Build processor request feature with a mock/stub notification, then implement borrower view later and connect them
  3. Option C - Single Ticket: Combine both into one larger ticket (#CP-101-301) if they're tightly coupled

Recommendation: Use Option B (Stub First) to maintain parallel development. Processor can build and test request feature with email-only notification, then borrower view enhances it later.

5. Testing Strategy Per Tier

Tier Testing Focus Acceptance Gate
Platform Can users view data? Is API contract stable? Does grid render correctly? Deploy to staging, have all roles log in and verify data loads
Role Scaffolding Can each role access their view? Are permissions correct? Does navigation work? Have actual users (processor, underwriter, borrower) test their views
Features Does the feature work end-to-end? Happy path + error cases User acceptance testing with real workflow scenarios

## Summary [One paragraph describing what this ticket accomplishes] ## Scope - [Bulleted list of what's included] - [Be specific: UI components, API endpoints, data models] ## Acceptance Criteria - ‚úì Given [context], when [action], then [expected result] - ‚úì Given [context], when [action], then [expected result] ## Out of Scope - [List what's explicitly NOT included to avoid scope creep] ## Data Mapping (if applicable) - Field: type, validation rules - API payload structure - External system references ## Dependencies - Depends On: #CP-XXX (must be complete first) - Blocks: #CP-YYY, #CP-ZZZ (these wait on this ticket) - Related: #CP-AAA (loosely connected) ## Technical Context (optional) - Reference to design mockups - Links to API documentation - Related decisions or architectural notes
