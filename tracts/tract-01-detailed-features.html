ðŸŽ¨ Feature #1: Main Grid View & Filtering

#CP-001
Platform Large Priority: High

Conditions Grid Platform - Read-Only Foundation

Summary

Create the foundational grid that displays all conditions for a loan, grouped by stage, with filtering, searching, and bulk selection capabilities. This establishes the core UI framework and API contracts.

ðŸŽ¯ Scope - Grid Components
  • Top Header Bar:
    • Loan number and borrower name display (e.g., "2024-001234 Conditions for Review - Johnson")
    • Loan status badge (e.g., "Ready for UW (from Processor after Approval)")
    • Current user role display (e.g., "Processor III")
  • Filter Pills (5 status filters):
    • "Active" with count badge
    • "Need to Request" with count badge
    • "Requested" with count badge
    • "Send Brw Requests" with count badge
    • "Review Docs" with count badge
  • Search Bar: "Search Conditions" textbox with icon
  • Action Buttons:
    • "Add Conditions" button (triggers modal)
    • "Update Loan Status" button
    • Selection count badge (e.g., "2" selected)
  • Grid Columns (7 columns):
    • Checkbox (for bulk selection)
    • CATEGORY
    • CONDITION DESCRIPTION
    • STATUS
    • CONDITION NOTES
    • DOC REQUESTS
    • DOCS
  • Stage Grouping:
    • Collapsible sections by stage (e.g., "Prior to Funding")
    • Count badge showing items in each stage
    • "Collapse All" button to minimize all sections
  • Condition Card (Row):
    • Checkbox for individual selection
    • Condition ID as clickable button (e.g., "CLSNG-001")
    • Expand/collapse icon
    • "Added By" label (e.g., "Added By System")
    • Condition name and description with colon separator
    • Type badge (INT or BRW)
    • Class label (e.g., "Class: Processor III")
    • Status badge (colored pill, e.g., "New" in blue)
    • Date display (e.g., "01/08/26")
    • "Set by" attribution
    • "Add a note" button
    • Doc requests section (shows count or "No requests")
    • "Add request" button
    • "Add Docs" button
âœ“ Acceptance Criteria
  1. Given a user loads the conditions portal, when the page renders, then all conditions are displayed grouped by stage with accurate counts
  2. Given conditions exist in multiple stages, when the page loads, then each stage section shows a collapsible header with condition count
  3. Given a user clicks a filter pill (e.g., "Active"), when the filter is applied, then only conditions matching that status are displayed and count badge updates
  4. Given a user types in the search box, when text is entered, then conditions are filtered in real-time by name, description, or ID
  5. Given a user clicks "Collapse All", when the button is clicked, then all stage sections collapse to show only headers
  6. Given a user clicks a condition checkbox, when selected, then the checkbox shows checked state and selection count badge increments
  7. Given multiple conditions are selected, when the user views the selection count, then it accurately displays number of selected items
  8. Given a user clicks on a condition ID (e.g., "CLSNG-001"), when clicked, then the edit condition modal opens with that condition's data pre-filled
  9. Given a condition has no document requests, when displayed, then it shows "No requests" in the DOC REQUESTS column
  10. Given a condition has status "New", when displayed, then the status badge appears in blue color
API Endpoints Required:
GET /api/loans/{loanId}/conditions
Returns: Array of conditions with full details, grouped by stage

GET /api/loans/{loanId}/conditions/summary
Returns: Filter counts (active, need_to_request, requested, etc.)
ðŸ”‘ Technical Notes
  • Grid uses virtualization for performance with large condition lists (100+ items)
  • Filter pills update counts via WebSocket or polling every 30 seconds
  • Search debounced at 300ms to avoid excessive API calls
  • Stage grouping persists collapse/expand state in localStorage
  • Checkbox selection state managed in React context for bulk operations

Add Document Request from Processor

Summary

Processor can create a document request for a specific condition, selecting from predefined document types and writing a custom description for the borrower. Request is linked to the condition and borrower is notified.

ðŸŽ¯ Modal Components (from POC)

Field Type Properties Validation

Request For Button/selector Pre-populated with borrower names At least one borrower must be selected
Description for Borrower Rich text editor Multiline, supports Bold, Italic, Underline, Link Max 1000 characters, shown as "0/1000"

Document Types (16 options in POC)

  • Appraisal
  • Asset Statements
  • Bank Statements
  • Employment Verification
  • Flood Insurance Binder
  • Gift Letter
  • Government ID
  • Hazard Insurance Binder
  • Homeowners Insurance
  • Mortgage Insurance
  • Pay Stubs
  • Purchase Agreement
  • Tax Returns
  • Title Insurance

Rich Text Editor Capabilities

  • Bold button: Applies bold formatting to selected text
  • Italic button: Applies italic formatting to selected text
  • Underline button: Applies underline formatting to selected text
  • Link button: Opens dialog to insert hyperlink with URL
  • Character counter: Shows current/max (e.g., "245/1000")

Button Behavior

  • "Cancel" button: Closes modal without saving, no API call
  • "Add" button: Disabled until Document Type is selected, enabled once required fields filled
âœ“ Acceptance Criteria
  1. Given a user clicks "Add request" on a condition, when the modal opens, then Document Type shows "Select", Request For is pre-filled with borrower name(s), and description is empty
  2. Given the modal is open, when Document Type has not been selected, then the "Add" button is disabled
  3. Given a user clicks the "Select" button for Document Type, when clicked, then a searchable dropdown appears with all 16 document types
  4. Given the document type dropdown is open, when a user types in the search box, then the list filters to matching document types in real-time
  5. Given a user selects a document type (e.g., "Bank Statements"), when selected, then the dropdown closes and the selected type is displayed
  6. Given a user types in the Description field, when typing, then the character counter updates (e.g., "125/1000")
  7. Given a user reaches 1000 characters in the Description field, when trying to type more, then input is blocked and counter shows "1000/1000"
  8. Given a user selects text and clicks the Bold button, when clicked, then the selected text is wrapped in bold HTML tags
  9. Given a user clicks the Link button, when clicked, then a dialog opens to enter URL and link text
  10. Given all required fields are filled, when the user clicks "Add", then the document request is created, linked to the condition, modal closes, and doc request appears in the condition's DOC REQUESTS column
  11. Given a document request is successfully created, when saved, then the borrower receives an email notification with the request details
API Endpoints Required:
POST /api/conditions/{conditionId}/document-requests
Body: { documentType, requestFor: [borrowerIds], descriptionForBorrower, requestedBy }
Returns: Created document request object with ID

POST /api/notifications/document-request
Body: { borrowerId, conditionId, documentRequestId, documentType, description }
Triggers: Email notification to borrower
ðŸ”‘ Technical Notes
  • Rich text editor uses a library like TipTap or Quill for formatting
  • HTML from rich text is sanitized on backend to prevent XSS
  • Document type dropdown uses fuzzy search (e.g., typing "bank" matches "Bank Statements")
  • Request For field supports multiple borrowers (APP1, APP2, etc.)
  • Condition status automatically updates to "Requested" when first doc request is created

Add Conditions from Template Library

Summary

Comprehensive modal for adding multiple conditions to a loan by selecting from a template library organized by categories. Supports custom lists, standard templates, and bulk addition of conditions.

ðŸŽ¯ Modal Layout (3-Panel Design from POC)

Left Panel: My Lists & Categories

  • "My Lists" section:
    • Displays user's saved condition lists
    • "Add New List" button to create custom list
    • Quick access to frequently used condition combinations
  • "Categories" section:
    • "All" button - Shows all available conditions
    • "Standard" button - Shows standard condition templates
    • "My Custom Templates" button - Shows user-created templates
    • 20+ category buttons (APP, DISC, CRED, INC, ASSET, PROP, MISC, CLSNG, TITLE, GOV, BOND, SP, CONDO, NEW CONST, etc.)
    • Clicking a category filters the Available Conditions panel

Middle Panel: Available Conditions

  • Header: "Available Conditions"
  • Search box: "Search All Conditions" with real-time filtering
  • "Custom" button: Allows user to create a custom condition from scratch
  • Condition list:
    • Checkboxes for selecting conditions
    • Condition name and description preview
    • Category badge (e.g., "CLSNG", "INC")
    • Loads dynamically based on selected category
    • Shows "Loading..." state when fetching
  • Action buttons:
    • "Add" button - Moves selected conditions to Current Conditions panel
    • "Remove" button - Removes selected conditions from Current Conditions panel
    • Both buttons disabled when no items selected

Right Panel: Current Conditions

  • Header: "Current Conditions"
  • Grid columns:
    • Checkbox (disabled until conditions added)
    • NO. - Condition number
    • DESCRIPTION - Condition name and description
    • REQUEST FOR - Which borrower(s) the condition applies to
  • Empty state: "No conditions added yet" when panel is empty
  • Populated state: Shows list of conditions ready to be added to loan

Modal Footer Buttons

  • "Cancel" button: Closes modal without adding conditions
  • "Add" button: Adds all conditions in Current Conditions panel to the loan
âœ“ Acceptance Criteria
  1. Given a user clicks "Add Conditions" on the main page, when the modal opens, then it displays three panels: My Lists & Categories (left), Available Conditions (middle), Current Conditions (right)
  2. Given the modal opens, when displayed, then the Available Conditions panel shows "Loading..." and then populates with all standard conditions
  3. Given a user clicks a category button (e.g., "INC"), when clicked, then the button is highlighted and Available Conditions panel filters to show only conditions in that category
  4. Given a user types in "Search All Conditions", when text is entered, then the Available Conditions list filters to show matching conditions by name or description
  5. Given a user selects checkboxes in Available Conditions, when one or more are checked, then the "Add" button becomes enabled
  6. Given conditions are selected and user clicks "Add" button, when clicked, then selected conditions move to the Current Conditions panel and checkboxes are cleared
  7. Given the Current Conditions panel has items, when displayed, then each condition shows its number, description, and REQUEST FOR field
  8. Given a user selects conditions in Current Conditions panel, when checkboxes are selected, then the "Remove" button becomes enabled
  9. Given user clicks "Remove" button, when clicked, then selected conditions are removed from Current Conditions panel
  10. Given Current Conditions panel has items and user clicks modal "Add" button, when clicked, then all conditions are added to the loan, modal closes, and main grid refreshes showing new conditions
  11. Given no conditions are in Current Conditions panel, when viewed, then it displays "No conditions added yet" message
API Endpoints Required:
GET /api/condition-templates
Query params: category (optional)
Returns: Array of condition templates

GET /api/condition-templates/search?q={query}
Returns: Filtered array of condition templates matching search

GET /api/users/{userId}/condition-lists
Returns: Array of user's custom condition lists

POST /api/loans/{loanId}/conditions/bulk
Body: { templateIds: [array of template IDs], requestFor: [borrower IDs] }
Returns: Array of created condition objects
ðŸ”‘ Technical Notes
  • Condition templates loaded lazily when category is selected to improve performance
  • Search is debounced at 300ms to avoid excessive API calls
  • Bulk creation API creates all conditions in a single transaction for data consistency
  • Category filter state persists when user switches between modal and main view
  • "My Lists" allows users to save frequently used condition combinations for quick access
  • Custom conditions created via "Custom" button are saved as user templates
